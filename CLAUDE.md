# Правила работы агента-исполнителя — pepperoni-api

Этот файл — рельсы для ИИ-агента (Cursor Agent / Claude Code), работающего над проектом.
Цель: агент несёт дисциплину проверки сам, чтобы человек подтверждал реже, а не чаще.
Положить как `CLAUDE.md` в корень репо (для Claude Code) и/или продублировать в
`.cursor/rules/` (для Cursor).

## 0. Главный принцип — ГОТОВО = СТРОКИ В origin/main, НЕ ПРОЗА
- «Сделано», «исправлено», «py_compile OK» — это НЕ доказательство. Доказательство —
  только закоммиченные и запушенные строки в `origin/main`.
- Любое изменение: внеси → закоммить → запушь → дай хеш коммита.
- Проверяй сам себя по коду ПОСЛЕ пуша: `git fetch && git checkout main && git pull`,
  затем `grep -nE` по полному файлу.
- Отсутствие чего-либо подтверждай **полным грепом** (`grep -c`), **НИКОГДА `head`** —
  усечённый вывод даёт ложноотрицательный результат (так уже была ошибка).

## 1. Диагноз — фактом, не догадкой
- Прежде чем чинить, докажи причину по коду/данным. Не предполагай «модель
  галлюцинирует» / «claim в промпте» — найди строку-источник.
- Если фикс основан на гипотезе — сперва подтверди гипотезу грепом. Премиса ложна →
  фикс бесполезен.

## 2. Корень, а не симптом; класс, а не один случай
- Чини источник (данные/промпт/конфиг), а не отдельный вывод.
- Если гейт поймал одно нарушение — ищи **весь класс** (другие суперлативы, другие
  пустые поля), а не только тот экземпляр, что всплыл.

## 3. Лимит итераций — стоп и эскалация
- Максимум ~3 попытки починить одно и то же. Не зашло — СТОП, опиши корень, эскалируй
  человеку. Не крутить петлю (это жжёт бюджет и время).
- Не «чини следующий симптом» бесконечно — если симптомы плодятся, проблема в корне
  (часто — среда, а не код).

## 4. Гейт священен — не ослаблять
- `data/invariants.json`, `page_reviewer.py`, `deploy_check.py` — принудительное ядро.
- НИКОГДА не ослабляй критерии рецензента/инварианты, чтобы «пропустить» контент.
- Карантин/hold = норма, а не сбой: лучше не опубликовать, чем опубликовать ложь.
- Нет утверждённого порога/значения → НЕ выдумывать. Эскалировать человеку.

## 5. Честность контента (конституция §4, устав сайта §5)
- Халяль — абсолют: никогда свинина-как-товар, никогда несуществующий сертификат.
- Никаких выдуманных отзывов/рейтингов.
- Никаких непроверяемых суперлативов и region-claim'ов («единственный/лучший/центр
  отрасли»). Конкретика вместо превосходных степеней.
- Решения о продуктовых claim'ах (правда/неправда) — НЕ автономно. Эскалировать владельцу.

## 6. Целостность среды (урок VPS)
- VPS обязан быть точным отражением `main` (`git rev-parse HEAD` == `origin/main`).
- Канал доставки кода — только `git pull` по SSH из `main`. `rsync` как канал кода —
  ЗАПРЕЩЁН (создаёт дрейф).
- Узел-производитель (что-то пушит сам) делает `git pull --rebase` ПЕРЕД commit,
  и НИКОГДА `reset --hard` (затрёт несохранённое).

## 7. Бюджет и необратимое
- Уважать дневной кап бюджета как предохранитель от разгона.
- Bulk/массовые операции — отдельный, утверждённый владельцем конверт, не дневной кап.
- Необратимые действия (массовая публикация, удаление, доступы) — подтверждение человека.

## 8. Что эскалировать человеку (не решать самому)
- Любое решение о ценностях/фактах (claim о продукте, рынке, статусе).
- Любое изменение порогов/инвариантов/конституции.
- Доступы, ключи, деплой-механизм.
- Любую ситуацию, где «правильно» зависит от знания, которого нет в данных.

## Резюме одной строкой
Автономия = задача + исполнение + **механический гейт** + лимит итераций + эскалация.
Где есть гейт — действуй сам. Где гейта нет — ты не гейт, эскалируй.

## 9. После SEO — индексирование по умолчанию
- После push SEO-изменений (title/canonical/301/sitemap/money-страницы) на VPS
  сразу: `bash scripts/nudge_google_after_seo.sh` (нужен `seo-agent.env`).
- Не ждать просьбы владельца. В отчёт — вывод скрипта.

## Handoff задачи
- Architect пишет **один** шаг в `instructions/next-task.md` (блок Current step).
- Worker читает `CLAUDE.md` + `instructions/next-task.md`, выполняет шаг, в Log даёт
  **хеш коммита + вывод grep/тестов**, не слово «готово».
